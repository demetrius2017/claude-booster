# Verify UI Flow

Автономная проверка UI flow. НЕ спрашивай пользователя — верифицируй сам.

URL или описание flow: $ARGUMENTS

## Протокол

### 1. Определи scope
- Если передан URL — проверяй его
- Если описание flow — определи URL и шаги сам из контекста проекта
- Если только что правил фронтенд-код — определи затронутые страницы

### 2. API слой (если применимо)
```
curl API endpoint → убедиться что бэкенд отдаёт данные
```

### 3. Browser verification (параллельно)
Используй Playwright MCP как основной:
- `browser_navigate` → открыть страницу
- `browser_snapshot` → accessibility tree (дёшево по токенам, структура данных)
- `browser_take_screenshot` → визуальное состояние
- `browser_console_messages` → JS ошибки
- `browser_network_requests` → failed requests

### 4. Проверки данных
Для каждого элемента flow:
- Таблицы: количество строк, наличие данных, не пустые ли ячейки
- Формы: поля доступны, validation работает
- Навигация: переходы между шагами работают
- Числа/суммы: сверить с API response если возможно
- Состояния: loading → data → empty state корректны

### 5. Multi-step flow (если flow из нескольких шагов)
Для каждого шага:
1. Выполни действие (click, fill, submit)
2. `browser_snapshot` → проверь что DOM изменился ожидаемо
3. `browser_take_screenshot` → визуальное подтверждение
4. Сверь данные на экране с ожиданием

### 6. Вердикт
Выдай таблицу:

| Шаг | Проверка | Результат | Evidence |
|-----|----------|-----------|----------|
| 1   | что проверял | PASS/FAIL | что увидел |

**PASS** — все проверки пройдены, evidence собран.
**FAIL** — root cause + что исправить. Не спрашивай разрешения — покажи проблему.

## Правила
- НЕ проси пользователя "проверьте в браузере" — ты и есть тестировщик
- Скриншот обязателен — это evidence
- Если snapshot показывает пустую таблицу/список — это FAIL, не "проверьте данные"
- Console errors = автоматический FAIL с деталями
- Network 4xx/5xx = автоматический FAIL
- Сравнивай данные в UI с API response когда возможно
