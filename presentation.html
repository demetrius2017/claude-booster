Память и пайплайн качества для Claude Code

Агент забывает всё к утру —
и сам ставит себе зачёт.

Claude Code не помнит ничего между сессиями и проверяет тот же код, который только что написал. Claude Booster чинит и то, и другое: даёт агенту накопительную память и пайплайн с гейтами, где ни одна модель не проверяет свою собственную работу. Одна команда python install.py — Mac или Linux.

Ни одна из этих проблем — не про интеллект модели. Это память, дисциплина и самопроверка — ровно тот слой, который чинит инструмент оркестрации.

объясняю
Ты описываешь стек, конвенции и режимы отказа каждую сессию заново.
изобретаю
Claude переписывает хелпер, который у тебя уже есть — потому что не сделал grep.
переспрашиваю
Те же уточняющие вопросы день за днём. «npm или pnpm?» — ты ответил вчера.
самооценка
Модель, которая написала код — она же его и одобряет. Коррелированные слепые зоны уходят в прод.
плоское время
Код, верный в T=0 и неверный в T+1 — async-колбэк, частичное исполнение, устаревший кэш.
старт с нуля
Каждый новый проект — с чистого листа. Добытые потом уроки не переносятся.
Три механизма

Что Booster на самом деле устанавливает

Не промпты. Движок памяти, самопроверяющийся пайплайн и учитывающий стоимость роутинг моделей — вшиты в Claude Code как правила и хуки.

01 · память

Темпорально-причинная память

Обычная память хранит факты. Booster хранит причинные цепочки: что пробовали → что случилось → что заключили → что осталось открытым. Детектор застреваний срабатывает, когда та же проблема всплывает без проверенного решения.

убивает «заново нахожу тот же баг каждую неделю»
02 · пайплайн

Семёрка — /go

Семь гейтов, кросс-провайдер. Сильная модель думает (критика дизайна, независимая верификация, ревью диффа); быстрая flat-fee модель печатает. Ни одна модель не ревьюит свой код, а вердикт — exit code теста.

убивает самопроверку и баги «плоского снимка»
03 · роутинг

Умный роутинг моделей

Haiku для поиска, Sonnet для кода, Opus для сложных рассуждений, flat-fee Codex где эквивалентно. Правильная модель под задачу — считается ежедневно по реальной телеметрии латентности и бюджета, а не по статичной догадке.

убивает оплату Opus-ставки за grep
/go  ·  пайплайн, который нельзя обмануть

Одна команда. Семь стадий. Ни одна модель не проверяет свою работу.

Когда делегируешь код через /go, хук физически блокирует любой обход. Стадии, отмеченные янтарём, идут на другом провайдере, чем автор — встроенная вторая пара глаз.

СТАДИЯ 6

Ревью диффа

✗ читает финальный дифф · кросс-провайдер
СТАДИЯ 7

Вердикт

exit code = вердикт → коммит

Почему это лучше одного умного агента: ~65% возвратов-к-коду — это неоднозначность контракта + пропущенные режимы отказа. Их ловят на этапе дизайна, а не более способным «печатником». Каждый запуск /go автоматически пишет исход в rework-KPI, поэтому эффективность пайплайна измеряется, а не предполагается.

18 slash-команд

Инструментарий — сгруппирован по задаче

В первом кластере ты живёшь ежедневно. Остальное берёшь, когда этого требует работа.

Ежедневный цикл — память между сессиями

/start
Восстанавливает контекст — handover, базу знаний, телеметрию — и планирует.
Старт каждой сессии вслепую.
/handover
Сохраняет структурный отчёт конца сессии с Goal, KPI и evidence.
Открытые нити испаряются в конце сессии.
/debt
Отслеживает и закрывает недоделки, переживающие ресет сессии.
«Потом поправлю» забывается мгновенно.

Пайплайн качества — ни одна модель не ставит себе зачёт

/go
Семистадийный кросс-провайдер пайплайн. Вердикт по exit code, без обхода.
Самопроверку и код, верный лишь в T=0.
/audit
Шесть параллельных агентов-линз + внешнее ревью, структурные вердикты.
Один ревьюер упускает то, что ловят шесть.
/code-review
Точечный проход после правки: дубли, оверинжиниринг, дрейф.
Переписывание хелпера, который уже есть.
/consilium
Дебаты 3–5 агентов + внешние модели, синтез в отчёт.
Одно мнение по рискованному решению.
/hackathon
N воркеров строят в изоляции; Судья оценивает всех одним сьютом.
Одну попытку на неопределённое решение.
/fable
Разовое read-only второе мнение от Fable 5.
Целый консилиум ради одной проверки.

Верификация — докажи, а не предполагай

/verify-after-edit
Прогоняет реальный UI-флоу после правки, а не только тесты.
«Компилируется» вместо «работает».
/verify-flow
Проверяет полный UI-флоу end-to-end.
Фикс проходит, а флоу, в котором он живёт, сломан.
/audit-trace
Трассирует один концепт по всем путям вычислений; расхождение = дефект.
Тихую порчу данных между путями кода.

Системное мышление и инфра — сначала пойми радиус поражения

/architecture
Генерит «печатную плату» — ARCHITECTURE.md + dep_manifest.json.
Правишь A и молча ломаешь B, C, D.
/phase
Ставит фазу воркфлоу; гейт требует «диагноз до починки».
Правку до понимания.
/delegate
Инспектирует гейт, заставляющий Lead делегировать код, а не писать самому.
Оркестратор тихо делает работу сам.
/lead
Ведёт supervised-воркера с квотой и детектором тишины.
Долгие автономные задачи дрейфуют без надзора.
/update
Автообновление Booster с GitHub, не выходя из сессии.
Трение апгрейда оставляет тебя на старых фиксах.
Одна команда, потом /start

Хватит каждое утро заново учить Claude Code одному и тому же.

Booster ставит набор правил, хуков и slash-команд в ~/.claude/. Твоя следующая сессия уже помнит, проверяет и аудитит себя.

git clone https://github.com/demetrius2017/claude-booster.git && cd claude-booster && python install.py
1 · install.py
2 · открой сессию → /start
3 · собери → /go
4 · закрой → /handover
