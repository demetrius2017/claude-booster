Three mechanisms

What Booster actually installs

Not prompts. A memory engine, a self-enforcing pipeline, and cost-aware model routing — wired into Claude Code as rules and hooks.

01 · memory

Temporal-causal memory

Standard memory stores facts. Booster stores causal chains: what was tried → what happened → what was concluded → what's still open. A stuck-loop detector fires when the same problem reappears without a verified resolution.

kills "re-discover the same bug every week"
02 · pipeline

The шестёрка — /go

Six gated stages, cross-provider. The strong model thinks (design critique, independent verification, diff review); the fast flat-fee model types. No model ever reviews its own code, and the verdict is a test's exit code.

kills self-review & flat-snapshot bugs
03 · routing

Smart model routing

Haiku for lookups, Sonnet for coding, Opus for hard reasoning, flat-fee Codex where it's equivalent. The right model for each task — measured daily from real latency and budget telemetry, not a static guess.

kills paying Opus rates for a grep
/go  ·  the pipeline you can't cheat

One command. Six stages. No model checks its own work.

When you delegate coding through /go, a hook physically blocks any shortcut. Stages marked amber run on a different provider than the author — a built-in second set of eyes.

STAGE 1

Flow Designer

maps failure modes & temporal traps — before any code
STAGE 2

Challenge

✗ attacks the plan · cross-provider
STAGE 3

Prototype Gate

proves the data hypothesis read-only
STAGE 4

Worker ‖ Verifier

verifier never sees the worker's code
STAGE 5

Test run

exit code = verdict
STAGE 6

Diff review

✗ reads the final diff · cross-provider

Why this beats a single smart agent: ~65% of returns-to-code are contract ambiguity + missed failure modes — caught at design time, not by a more capable typist. Every /go run auto-records its outcome to a rework KPI, so the pipeline's effectiveness is measured, not assumed.

Quality pipeline — no model grades its own homework
