## Phase 2d — what landed

### New helpers in `~/.claude/scripts/rolling_memory.py`

- **`get_readonly_connection()`** — opens `DB_PATH` via `file:...?mode=ro`
  URI so the connect call does NOT create an empty file on a missing DB.
  Enforces the "no DB writes" contract at the filesystem layer, not just
  the SQL layer.
- **`_category_from_scope(scope)`** — derives the
  `index_reports._project_category` mirror from a scope path. Strategy:
  normalize via `expanduser().resolve()`, anchor to `~/Projects` via
  `relative_to()`, then walk ancestors UP until finding a dir that owns
  `reports/` or `audits/`. Returns `None` gracefully for out-of-Projects
  scopes and edge cases. Replaces an earlier greedy-descent version that
  round 1 caught as a HIGH bug.
- **`build_start_context(scope, query=None, limit=10)`** — category-biased
  SQL over consilium/audit rows, optional FTS5 join when `query` is set,
  4 ordering branches, distinct error messages for FTS5 syntax errors
  vs missing-table/file vs other failures. Reads via
  `get_readonly_connection()`. 10 LOC bullet-list renderer with `*` for
  in-project rows and `-` for cross-project.

### New CLI subcommand

```
rolling_memory.py start-context [--scope PATH] [--query "terms"] [--limit 10]
```

Dispatcher gate:
```python
if args.cmd != "start-context":
    init_db()
```
keeps `start-context` strictly read-only (R1-MED fix). Other subcommands
still call `init_db()` unchanged.

`--scope` defaults to `os.getcwd()` wrapped in `try/except OSError` (R2-LOW
fix — handles deleted cwd gracefully).

### `/start` rule prose change

`~/.claude/rules/commands.md` step 2 now reads:

```
python ~/.claude/scripts/rolling_memory.py start-context --scope "$(git rev-parse --show-toplevel 2>/dev/null || pwd)"
```

with an optional `--query "<keywords>"` for topic-driven search and a
self-heal note directing the user to `index_reports.py` if the DB is
empty. Git-toplevel resolution ensures the scope is the project root even
when Claude is launched from a subdirectory; quoted to survive paths with
spaces.

## GPT-5.4 round 2 findings → fixes

Round 1's continuation ID expired at the 3-hour mark. Opened a fresh
continuation with a self-contained brief covering all 4 R1 fixes + the
6 edge cases I pre-exercised. GPT-5.4 focused on what I hadn't already
ruled out and returned 2 new findings:

| # | Sev | Issue | Fix |
|---|---|---|---|
| R2-MED | MED | "No DB writes" contract was only true at the SQL layer. `sqlite3.connect(str(DB_PATH))` is a stdlib WRITE primitive on missing paths — creates an empty 0-byte file. A `/start` on a fresh machine would silently create the file, then fail the SELECT, then print the friendly bootstrap message. Contract-vs-reality drift. | New `get_readonly_connection()` via `file:{DB_PATH}?mode=ro` URI. `build_start_context` switched to it. "Unable to open database file" mapped to the same "DB not initialized" message. Verified: missing DB → friendly message → no file created (`tmpdb.exists() == False`). |
| R2-LOW | LOW | `scope = args.scope or os.getcwd()` — `os.getcwd()` raises `FileNotFoundError` if cwd was deleted out from under the shell. CLI crash with unhelpful traceback. | Wrap in `try/except OSError` → fall back to `scope=None` so `_category_from_scope` returns no bias and the CLI produces unbiased output instead of crashing. |
