## Problems / open items (deferred to v1.1)

1. **R7 — Code-only First-step bodies.** Booster handover First-step often
   pure bash → tokens=[] → all bash-only handovers hash identically. Detector
   fires on real "code-only handoff" pattern but loses verify_gate FP signal.
   v1.1 fix: extend body extraction to Next-step/Open items sections.

2. **R8 — Nested-JSON evidence.** `has_pass_for_hash` flat-block regex
   correctly enforces same-block requirement but false-negatives on nested
   `{"verified": {"status": "pass", "stuck_hash": "..."}}`. Current verify-gate
   template is flat; if it changes to nested, clearance silently breaks.
   v1.1: small JSON tokenizer for nested case OR explicit flat-block contract
   in template comment.

3. **R9 — PAL minor deferrals (4 items, ~30 LOC cleanup).**

4. **R10 — Window-size mismatch** (observed at MERGE-phase smoke):
   telemetry uses 10-handover window (inherited from cadence/evidence
   signals), rolling_memory uses 5-handover per spec §D2. On borderline
   counts, telemetry=None, rolling_memory=⚠. Both individually correct;
   user sees one ⚠ and one ✓. v1.1: harmonize, choice eager-vs-conservative
   deferred to 2026-05-13 review.

5. **CLI: `--json` is top-level flag.** `--json --stuck-check` order matters in
   argparse — `--json` MUST precede subcommand. Pre-existing CLI behavior,
   not introduced by this work, but worth flagging for future doc/users.
