## Summary

Not a Booster-code session — an environment fix. Diagnosed from runtime state (not
docs): Dmitry runs Claude Code inside tmux 3.6a inside iTerm2 with `tui: "fullscreen"`.

Root cause was a stack of three:
1. **Fullscreen renderer** draws to the *alternate screen buffer*, so the conversation
   is NOT in tmux's native scrollback. Scrolling via tmux copy-mode/wheel hits an
   empty/corrupt scrollback → the "обрывается" symptom (Claude Code bug #58364).
2. **`TERM=screen-256color` + `default-terminal "screen-256color"`** — stale terminfo
   that poorly describes alternate-screen + synchronized output → render breakage.
3. **Missing `focus-events on`** — the exact thing the startup hint on Dmitry's
   screenshot complained about; without it Claude doesn't redraw on focus change.

Verified-safe before editing: `infocmp tmux-256color` exists on this macOS box (setting
a missing terminfo would have broken tmux entirely — the guard that mattered).

## Changes — `~/.tmux.conf` (outside Booster repo, not git-tracked)

```
set -g default-terminal "tmux-256color"          # was screen-256color
set -g focus-events on                            # added
set -sa terminal-features ",xterm-256color:RGB"   # added — truecolor passthrough to iTerm2
```

Config reloaded via `tmux source-file ~/.tmux.conf`. All three confirmed with `tmux show`.
