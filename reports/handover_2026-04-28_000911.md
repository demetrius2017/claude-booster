# Handover — 2026-04-28

## Summary
Расширил `~/.claude/scripts/delegate_gate.py`: добавил read-only Bash classifier `_bash_is_readonly_diag(cmd)`. Lead теперь делает диагностический Bash (ssh+ps/grep, git status/log/diff, docker ps/logs, kubectl get/logs, helm list, systemctl status, ls/cat/grep/find, curl GET, journalctl) **без расхода** delegation budget. Write-Bash остаётся под budget=1: `git push/commit/merge/rebase/reset`, `docker run/build/rm/restart`, `kubectl apply/delete/edit`, `pip/npm/yarn install`, `helm install/upgrade`, `curl -X POST/PUT/DELETE/PATCH`, `tee`, redirect `>file` к не-`/dev/null|/tmp/*`.

Edit/Write/NotebookEdit на код по-прежнему счётчик; на `/reports/`, `/docs/`, `*.md`, `.claude/` — уже было в `ALLOWLIST_PATHS`.

## Why
Жалоба пользователя: "bash команды пытается делегировать и ответа не получает". В цепочке расследования (ssh на droplet → ps → grep → выясняем кто держит DuckDB lock) первая Bash проходит, вторая ловит `direct-action budget exhausted (2/1)`, дальше Lead уходит в делегирование тривиальных диагностических команд агенту, агент висит.

Профессорская трактовка правила pipeline.md «Lead orchestrates, doesn't do work»: Lead не должен **писать код** или **делать write-операции**. Read-only диагностика — часть оркестрации (recon перед спавном агента), не подмена.

## What changed
**Один файл**, **outside этого репо**: `~/.claude/scripts/delegate_gate.py` (~+250 строк).

Добавлены:
- `READ_ONLY_BINARIES` — set чистых read-only утилит (ls/cat/grep/ps/find/awk/sed/jq/dig/ping/...).
- `READ_ONLY_SUBCOMMANDS: dict[str, set[str]|None]` — writeful бинари с whitelist-ом сабкоманд: git (status|log|diff|...), docker (ps|logs|inspect|compose), kubectl (get|describe|logs|...), helm (list|status|get|...), systemctl (status|show|cat|...), pip (list|show|...), npm (ls|view|...), terraform (validate|show|...). `journalctl` — `None` (все формы read-only).
- `_CURL_WRITE_FLAGS` / `_WGET_WRITE_FLAGS` — regex для write-маркеров (-X POST/PUT/DELETE/PATCH, -d, --data*, -F, --form, -T, --upload-file).
- `_REDIRECT_RE` + `_SAFE_REDIRECT_TARGETS` — `>file` сейф только для `/dev/null|/dev/stdout|/dev/stderr|/dev/tty|&N|/tmp/*`.
- `_SSH_WRITE_TOKENS` — heuristic regex для write-операций внутри ssh-команды (rm/mv/dd/mkfs/reboot/apt install/yum install/systemctl start/stop/restart/git push/commit/docker run/build/kubectl apply/delete/helm install/chmod/iptables/tee/echo>file).
- `_quote_aware_split(s, seps)` — split по `;`/`&&`/`||`/`|`, игнорируя separators внутри `'...'` или `"..."`. Регулярка не справлялась — `ssh host 'ps | grep'` рвался на куски.
- `_strip_leading_noops` — убирает `time`/`nohup`/`sudo`/`env KEY=val`/`exec` префиксы.
- `_segment_is_readonly(seg)` — классификация одного pipeline-сегмента.
- `_bash_is_readonly_diag(cmd)` — публичная функция: сегменты по `;`/`&&`/`||`, потом по `|`, каждый должен быть read-only. Backticks и process substitution → конс ервативно False.

В `main()`:

```python
        ...  # already existed
    if _bash_is_readonly_diag(cmd):

```

Module docstring обновлён — описана новая категория "Read-only Bash".

## Bugs caught during testing
1. **Trailing `|` в `_SSH_WRITE_TOKENS` regex** создавал empty alternative `(rm|mv|...|tee\b|)` и матчил пустую строку в позиции 0 для любой команды. Симптом: ВСЕ ssh-команды классифицировались как action. Fix: убрал последний `|` после `tee\b`.

2. **`_PIPE_SPLIT_RE.split` рвал ssh-команду внутри одинарных кавычек.** `ssh host 'ps -ef | grep python'` распадался на `["ssh host 'ps -ef", "grep python'"]`, классификация ломалась. Fix: написал `_quote_aware_split` — ручной парсер с tracking quote-state.

3. **`'rm` в `ssh host 'rm ...'` не матчился `_SSH_WRITE_TOKENS`** потому что boundary char class `[\s|;&]` не включал `'` или `"`. После открывающей кавычки `rm` не считался "началом слова". Fix: добавил `'\"` в boundary → `[\s|;&'\"]`.
