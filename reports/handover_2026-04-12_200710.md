## Pivot from the original plan

The plan file (`ancient-drifting-kazoo.md`) originally proposed using an FTS5 **column filter** (e.g. `scope:"..."`) to restrict queries. During Gate 2 I inspected real-world scope values in the DB and found them to all be filesystem paths like `/Users/dmitrijnazarov/Projects/AINEWS/ainews-sre-agent`. The default FTS5 `unicode61` tokenizer shreds those on `/`, `_`, `-`, `.` — so a column filter for `scope:"..."` would match on arbitrary path-component tokens (`users`, `projects`, `ai`, etc.) and silently leak cross-project rows.

**Pivot decision**: keep the v3 FTS scope column for schema parity/future-proofing, but do all scope filtering via the **joined `agent_memory.scope` column** (same pattern `_find_similar` already uses). This matches the plan's own Risks section fallback ("if FTS5 rejects it for scope values with dots/underscores, fall back to an `AND scope IN (...)` SQL clause"). GPT-5.4 confirmed this was the correct call.

## External AUDIT findings (GPT-5.4, all fixed)

### HIGH-1 — v2→v3 migration was not atomic

**Issue.** My initial `init_db()` implementation sequenced `DROP TRIGGER / DROP TABLE → commit → CREATE → commit → REBUILD → commit`. During the gap between DROP and CREATE, a concurrent reader calling `search()` or `_find_similar()` would see a missing/empty `agent_memory_fts` and silently return `[]`, logging an exception inside the tool but giving no user-facing signal. Same class of bug as session 3's "silent data loss" failure modes.

**Fix.** Build the entire v2→v3 migration as a single script string wrapped in explicit `BEGIN IMMEDIATE; ... COMMIT;` and run it via `conn.executescript()`. Python's sqlite3 `executescript` honours explicit transaction boundaries written into the script text. Now the drop, recreate, rebuild, and `PRAGMA user_version` bump all commit together — no reader ever observes a half-migrated state.

### HIGH-2 — `init_db()` did not self-heal missing FTS on a v3 DB

**Issue.** The rebuild flag only fired on `version < 3`. If a v3 DB were restored from a backup that lacked `agent_memory_fts`, or repaired manually, `init_db()` would silently leave the FTS empty because `version=3` → skip rebuild.

**Fix.** Before running the CREATE scripts, capture `fts_exists_before = ...sqlite_master WHERE name='agent_memory_fts'`. Decision: `should_rebuild_fts = need_v3_migration or (table_exists and not fts_exists_before)`. Verified by manually dropping FTS+triggers on a v3 DB and confirming the next `init_db()` restores FTS with full row parity.

### HIGH-3 — `index_reports.py` crashed on non-string frontmatter fields

**Issue.** `build_row()` did `(fm.get("type") or "").strip().lower()` — if a YAML frontmatter field happened to be a list, int, or dict (e.g. `type: [audit]` or `description: 42`), `.strip()` would raise `AttributeError` inside `build_row()`, bypassing the per-file `try/except` inside `index_all()` and aborting the whole indexing run. One malformed report could poison the entire index.

**Fix.** Added `_string_field(value) -> str` helper that returns `value.strip() if isinstance(value, str) else ""`. Replaced all three frontmatter field accesses. Also wrapped the `build_row()` call inside `index_all()` in a per-file `try/except` so any residual parse failure becomes a per-file `[ERR]` line with no effect on later files. Regression test: constructed a synthetic malformed file with list/int fields → `build_row()` returns valid kwargs with `memory_type='consilium'` inferred from filename.

### MED-1 — Scoped `search()` let global rows crowd out exact-scope rows

**Issue.** When called with `scope='/proj/A'` + `include_global=True`, the query was `(am.scope = ? OR am.scope = 'global') ORDER BY rank LIMIT ?`. A highly-ranked global row could consume the LIMIT and hide relevant project-local rows — exactly the failure mode `_find_similar` already guards against with `ORDER BY CASE WHEN am.scope = ? THEN 0 ELSE 1 END, rank`.

**Fix.** Added the same `CASE` prefix to the ORDER BY when `exact_scope_first=True`. Regression test: seeded a global row with `alpha * 20` (strong rank) and a `/proj/A` row with `alpha weak mention` (weak rank) → the scoped search now returns `/proj/A` first, matching the `_find_similar` pattern.

### MED-2 — Nested-project category inference was wrong

**Issue.** `_project_category()` returned `parts[0]` — so a report at `~/Projects/AINEWS/ainews-sre-agent/reports/audit_foo.md` was tagged `AINEWS` even though `ainews-sre-agent` is its own codebase. Semantically lumped separate projects together for the `category` field.

**Fix.** Walk `rel.parts` looking for the marker `reports` or `audits`; return the segment immediately before it. For `~/Projects/PnL_Tracker/audits/<topic>/audit_report.md`, `audits` is at index 1 so we return `PnL_Tracker` (correct — `<topic>` is a topic folder, not a sub-project). Verified 4 cases:

| Path | Before fix | After fix |
|---|---|---|
| `AINEWS/reports/...` | `AINEWS` | `AINEWS` |
| `AINEWS/ainews-sre-agent/reports/...` | `AINEWS` | `ainews-sre-agent` ✅ |
| `AINEWS/tnews_webapp/reports/...` | `AINEWS` | `tnews_webapp` ✅ |
| `PnL_Tracker/audits/<topic>/audit_report.md` | `PnL_Tracker` | `PnL_Tracker` |

Re-indexing after the fix changed category distribution from `{AINEWS:9}` to `{AINEWS:6, ainews-sre-agent:3}`.

### LOW-1 — Stale SCHEMA_VERSION comment

**Issue.** The `SCHEMA_VERSION = 3` comment described the v2→v3 migration as "scope filtering happens inside FTS MATCH" — which was my original plan before I pivoted to JOIN-level filtering. Misleading for future maintainers.

**Fix.** Rewrote the comment to document the pivot explicitly: "`scope` column added for schema parity/future-proofing; scope filtering in `search()` and `_find_similar()` still runs on the joined base table because `unicode61` tokenization shreds filesystem-path scopes."

## Process Lessons (candidates for `institutional.md`)

1. **Always query real data before relying on "expected" value shapes.** My plan assumed scope values would be short identifiers suitable for FTS5 column filtering. They're filesystem paths. One `sqlite3 "SELECT DISTINCT scope..."` in Gate 2 flipped the whole design decision. Verify input shapes against the real DB before committing to a plan.
2. **`executescript()` respects explicit BEGIN/COMMIT in the script text.** This is how you do atomic DDL+DML migrations in Python's sqlite3 without manually splitting statements. `executescript` implicit-commits any pending transaction first, then runs the script as-is.
3. **The AUDIT phase is not optional (still).** GPT-5.4 found 3 HIGH + 2 MED + 1 LOW in one pass, including two HIGH issues I would not have caught in my own post-implementation review. Session 3's lesson continues to hold: external review round-trip is mandatory, not boilerplate.
4. **FTS self-heal is load-bearing for backup recovery.** If you rebuild the FTS only during version migration, a backup restore that misses the FTS table silently leaves the search index empty with no error. Self-heal on every `init_db()` based on `sqlite_master` probe, not just on version transitions.
