### Backup tooling (UNPLANNED — DONE)

Dmitry pushed back on the practice of running raw `cp ~/.claude/rolling_memory.db ...bak_$(date +...)` inside session work, on the grounds that backups accumulate without retention. Inventory at session start: 4 stale `.bak*` files (`bak`, `bak_phase2b`, `bak_phase2c`, `bak_cleanup_171914`) plus the `.bak_cleanup_171914` I created earlier in this session = 5 total, ≈4.8 MB.

Spec he set: **maximum 2 backups, prune happens at create time** (not as a separate cleanup mode). Quote:

> «нужен роллинг бэкапов не более 2х … это должно делаться в момент создания бэкапа уставлять только новый и предыдущий остальные бэкапы этой базы удалять»

Built `~/.claude/scripts/backup_rolling_memory.py` (~110 LOC):

- Uses `sqlite3.Connection.backup()` (native SQLite backup API) instead of `shutil.copy` — safe on a live DB with concurrent writers, atomic from SQLite's perspective.
- Glob `{db_basename}.bak*` matches both timestamped backups and the bare `.bak` variant, so legacy files get rotated out.
- Sort by **mtime descending** (not filename), since naming schemes vary across sessions.
- Pruning is in-process inside `create_backup()`; there is intentionally no "cleanup-only" CLI mode.
- CLI flags: `--db PATH` (default `~/.claude/rolling_memory.db`), `--retention N` (default 2). Reusable for any SQLite DB.

Ran it once. Result: created `rolling_memory.db.bak_20260413_174337`, pruned 3 older backups, retained 2 (the new one + `bak_cleanup_171914`). `PRAGMA integrity_check` on the new backup → `ok`. Active row count in backup matches live DB (62 — that's `61 + 1` from a `PostToolUse` auto-captured row added during this session).

Saved a `feedback` memory at `~/.claude/projects/-Users-dmitrijnazarov-Projects-Claude-Booster/memory/feedback_rolling_memory_backup.md` documenting the policy, and indexed it in the project `MEMORY.md`. Future sessions will route through the script instead of reaching for `cp`.
