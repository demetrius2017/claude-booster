## Summary

The session opened on the planned `install.py` Codex-bridge integration but
**pivoted immediately** when Dmitry surfaced a live blocker: in a Codex session
on `~/Projects/CRM_AI`, `git add` failed with
`fatal: Unable to create '.git/index.lock': Operation not permitted`, and the
Codex agent had gone down a fragile GitHub-API workaround (creating a remote
branch and uploading file blobs without local `.git`).

Root cause (verified against Codex docs, not memory): Codex CLI 0.138.0 runs in
`sandbox_mode = "workspace-write"`, which **intentionally mounts `.git/`
read-only** even in a `trusted` project. There is **no granular option** in
`[sandbox_workspace_write]` to re-open only `.git` — the table exposes only
`writable_roots`, `network_access`, `exclude_slash_tmp`, `exclude_tmpdir_env_var`,
and `.git` is protected even inside `writable_roots`.

Dmitry chose **Variant A**: drop the FS sandbox globally so Codex matches Claude
Code's effective freedom on this trusted machine. Applied in
`~/.codex/config.toml`: `sandbox_mode = "workspace-write"` →
`"danger-full-access"`, with an updated rationale comment. TOML re-parses clean.

The "partial" branch turned out **not** to be junk: `codex/finmodel-r1-r2` on
origin contains 4 coherent commits (+161 lines: R1/R2 financial-model rows, a
Python usecase change, and a 92-line test). It was **left intact** — deleting it
was rejected because it holds real, deliverable work. A PR from it is the natural
next step (not done — externally-visible action, awaiting Dmitry's go).
