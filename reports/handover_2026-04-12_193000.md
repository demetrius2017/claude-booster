## Additional AUDIT phase fixes (GPT-5.4 found)

After all 7 gates passed, I invoked `mcp__pal__codereview` (model `gpt-5.4`, thinking=high) as per the pipeline's AUDIT phase. GPT-5.4 independently identified two MED issues I had not caught in my own post-implementation self-review:

### AUDIT-1 (MED) — R5 bridge-row visited leak

**Issue.** In `_cluster_memories`, when post-BFS seed re-verification rejects a bridge member, the rejected row stayed marked `visited=True` and was **silently lost** from all subsequent clustering. Pathological case: `A ~ B ~ C` where `A !~ C`. BFS starting from A builds `[A, B, C]` via B. Re-verification rejects C (because `A !~ C`). Result: cluster `[A, B]`, but C is now flagged visited and never gets a chance to seed its own cluster. **Silent data loss** the moment real consolidation runs on a corpus with chain-structured content.

**Fix.** In the post-BFS loop, on rejection: `visited[idx] = False`. Rejected bridges are returned to the unvisited pool and can seed a later cluster.

**Regression test (in-memory).** Constructed `a_kw/b_kw/c_kw` so that `A~B` and `B~C` pass the cluster gate (`subset=0.83, jaccard=0.45, shared=5`) but `A~C` fails (`shared=0`). Pre-fix behavior would have emitted `[[A,B]]` and lost C entirely. Post-fix: `[[A,B], [C]]` — C is preserved as its own singleton (which downstream filter drops, but not silently dropped from the corpus).

### AUDIT-2 (MED) — Lazy Anthropic client init happens AFTER DB mutations

**Issue.** My initial fix refactored `_get_anthropic_client()` to lazy init inside the cluster-synthesis loop, so that a 0-cluster real run wouldn't require `ANTHROPIC_API_KEY`. GPT-5.4 pointed out this reordered failure mode: `recall(touch_access=True)` runs **before** the client preflight, so a missing API key in a non-dry-run now fails **after** `last_accessed_at` has been mutated. The user sees `RuntimeError` but the DB is already dirty. The original eager init had this property right; my lazy fix broke it.

**Fix.** Eager preflight at the top of `consolidate()`: `if not dry_run: client = _get_anthropic_client()`. Drops the lazy-init block inside the synthesis loop. Cost: one `Anthropic()` object construction per real run even when no clusters exist. Acceptable tradeoff for correctness. Verified: running a real consolidation with no `ANTHROPIC_API_KEY` now raises the `RuntimeError` **before** any `recall()` call, leaving the DB byte-identical (sha256 verified).

Both fixes applied in-session; Gate 1 + Gate 6 re-verified clean after the fixes.

## Process Lessons (worth promoting to `institutional.md`?)

1. **Code reviewer's own review isn't enough — always round-trip through an external reviewer.** My post-implementation self-review caught 5 items but missed the R5 bridge-row visited leak (MED, silent data loss) and the lazy-init-after-recall ordering (MED, corrupts DB on failed real run). GPT-5.4 found both in one pass. The pipeline's AUDIT phase is not optional boilerplate.
2. **Regression tests need to actually trigger the bug they test for.** My first R5 bridge test used sets that didn't clear the cluster gate at all, so the code path being tested was never executed. Principle: *if the fix didn't run, the test didn't test*. Verify by forcing the pre-fix path to fail before declaring the post-fix path passes.
3. **Schema migrations that reference new columns in a CREATE INDEX must run the ALTER first, not the CREATE script first.** `CREATE TABLE IF NOT EXISTS` is a no-op on existing tables, so a new column declared inside it never materializes on pre-existing DBs. Explicit ALTER first, then the idempotent CREATE scripts. This is now institutional knowledge for the Claude_Booster project.
